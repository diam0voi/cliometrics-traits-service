В ходе анализа дампов и требований геймдизайнеров выявлено критическое несовпадение идентификаторов:
1. **Legacy-система** оперирует целочисленными ID (INT).
2. **Геймдизайнеры** в JSON-вишлистах используют человекочитаемые имена ("Joseph Stalin") или технические слаги ("USA_FDR").
3. **Бизнес-требования** обязуют использовать UUID для новой базы.

Семён не мог связать данные, так как в новой базе отсутствовала таблица-посредник, способная сопоставить текстовый "ключ" из JSON с числовым ID из старой системы.

## Предложенное решение
Внедрена таблица `LEADERS_MAPPING`, которая выполняет роль "реестра соответствий":
- **gd_key**: Сюда записываются значения из поля `applies_to_leaders` (например, "Hirohito").
- **legacy_id**: Сюда записывается старый ID из iOS-клиента.
- **internal_id**: Генерируется UUID, который становится основным ключом для всей новой системы трейтов.

## Обработка параметров времени
Для поддержки значений `active_period: "ALWAYS"` из JSON, в базе предусмотрен булев флаг `is_always`. Это избавляет от необходимости парсить смешанные типы данных (строка/объект) на стороне Core-движка и позволяет использовать простую SQL-логику фильтрации.
